1. Introduction

TaskHive is a production-ready, multi-role service marketplace platform built with React 18, Vite, TypeScript, Tailwind CSS, and a managed Supabase (Lovable Cloud) backend. It supports four user roles — Customer, Service Provider, Employer, and Admin — with end-to-end flows for bookings, job posting, applications, payments, reviews, messaging, and verification.

  • Fully responsive PWA with installable mobile experience
  • Role-based dashboards and protected routes
  • Realtime messaging, notifications, and live updates
  • Provider subscriptions, sponsored services, and pay-per-job posting
  • Admin suite: users, verifications, payments, jobs, analytics, CMS, banners, coupons

Who is it for?

  • Marketplace operators launching a Fiverr / TaskRabbit / Upwork style vertical.
  • Franchises & agencies that need per-tenant branding and isolated data.
  • Solo founders who want a production baseline (auth, payments, admin) on day one.

2. Architecture Overview

TaskHive is a single-page React app talking to a managed Postgres via PostgREST, with Deno edge functions for privileged / third-party work and Supabase Realtime for live channels.

┌────────────────────┐        ┌───────────────────────────────┐
│  React 18 + Vite   │◄──────►│  Supabase (Lovable Cloud)     │
│  shadcn / Tailwind │  HTTPS │  Postgres · Auth · Storage    │
│  React Query cache │        │  Realtime · Edge Functions    │
└─────────┬──────────┘        └──────────────┬────────────────┘
          │  Web Push / SW                    │  pg_cron jobs
          ▼                                   ▼
   Service Worker                    Scheduled tasks
   (Workbox precache)                (expire, reminders)

Request lifecycle

  1. Component calls a hook (useBookings, useServices…).
  2. Hook uses the shared supabase client from src/integrations/supabase/client.ts.
  3. PostgREST enforces RLS using the caller's JWT.
  4. Results are cached by React Query and invalidated by realtime channels or mutations.

Privileged work

Anything requiring the service role, third-party APIs, or long-running logic goes into an edge function under supabase/functions/<name>/index.ts. Never call the service role from the browser.

4. Environment Setup (.env)

Three public variables are required at build time:

VITE_SUPABASE_URL="https://<project-ref>.supabase.co"
VITE_SUPABASE_PUBLISHABLE_KEY="<anon key>"
VITE_SUPABASE_PROJECT_ID="<project-ref>"

Server-side secrets (edge functions only) — set via the Lovable secrets manager:

  • SUPABASE_SERVICE_ROLE_KEY — admin operations
  • LOVABLE_API_KEY — AI gateway (smart matching, descriptions)
  • DEMO_RESET_SECRET — gated demo data reset endpoint

Only variables prefixed with VITE_ are exposed to the browser bundle. Anything else stays server-side and is only readable inside edge functions via Deno.env.get().

Optional integrations

  • STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET — card payments & webhooks
  • PAYPAL_CLIENT_ID, PAYPAL_SECRET — PayPal payments
  • RESEND_API_KEY — transactional email
  • GOOGLE_MAPS_API_KEY — geocoding & place search
  • VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, VAPID_SUBJECT — Web Push

6. Backend & Data Model

The client is pre-wired at src/integrations/supabase/client.ts. Schema is managed through versioned migrations under supabase/migrations/.

Core tables (57 total)

  • Identity: profiles, user_roles, provider_staff, employer_profiles
  • Marketplace: vendor_services, tasks, task_proposals, bookings, reviews
  • Money: wallets, wallet_transactions, payment_transactions, invoices, withdrawals, refund_requests, category_commissions
  • Engagement: conversations, messages, notifications, favorites, saved_jobs
  • CMS / Admin: cms_pages, homepage_content, banners, advertisements, platform_settings, promo_coupons, email_templates
  • Trust & Safety: vendor_verifications, disputes, content_reports, admin_audit_log

Key relationships

auth.users ─┬─ profiles (1:1)
            ├─ user_roles (1:N)   role ∈ {client,provider,provider_staff,employer,admin}
            └─ wallets (1:1)

profiles(provider) ─┬─ vendor_services (1:N) ─┬─ bookings (1:N) ─┬─ reviews (1:1)
                    │                          │                  └─ payment_transactions
                    ├─ vendor_verifications    └─ favorites
                    └─ provider_staff (1:N)

profiles(employer) ── tasks (1:N) ── task_proposals (1:N) ── bookings

bookings ── invoices (1:1) ── withdrawals (via wallet_transactions)

Row-Level Security

Every public table has RLS enabled. Policies follow four patterns:

  • Owner-only: user_id = auth.uid() (wallets, favorites, saved_jobs)
  • Public read, owner write: vendor_services, reviews, profiles
  • Party read: both parties in a booking / conversation can read
  • Admin bypass: has_role(auth.uid(), 'admin') via SECURITY DEFINER

7. Authentication System

Email/password authentication powered by Supabase Auth with optional Google OAuth. Sessions persist via localStorage with auto-refresh.

  • Sign up → handle_new_user() trigger creates a profile + assigns role from user_metadata.role.
  • Sign in → log_login_event() writes to admin_audit_log.
  • Soft-delete via profiles.deleted_at blocks sign-in and signs the user out.
  • Profile completion gate (/complete-profile) for first-time users.

Flows

  1. Email sign-up → confirmation email → callback at /auth/callback → role assigned → onboarding wizard.
  2. Google OAuth → redirect_uri = ${window.location.origin}/auth/callback; Supabase provider must be enabled.
  3. Password reset → supabase.auth.resetPasswordForEmail() with redirect to /reset-password.
  4. Session hydration → useAuth() subscribes to onAuthStateChange and gates all protected routes.

Session lifecycle

Tokens auto-refresh every ~55 min. On SIGNED_OUT the app clears React Query cache and redirects to /login. On TOKEN_REFRESHED the new JWT propagates to realtime channels automatically.

Common pitfalls

  • Never call auth methods inside a Supabase auth callback synchronously — always setTimeout(..., 0) to avoid deadlocks.
  • Do not point OAuth redirect_uri at a protected route; use /auth/callback and navigate after session hydration.

9. Payments & Wallet

Money flows through a ledger-style wallet_transactions table. Every debit has a matching credit, and balances are computed from the ledger — never stored raw.

Providers supported

  • Stripe — cards, 3DS, webhooks for capture & refund
  • PayPal — Orders v2 API, capture on approval
  • Wallet — internal credit for refunds and promo balance

Fee math

gross         = service_price × qty
deposit       = gross × 0.25          // 25% up front
platform_fee  = gross × commission_rate  // per-category, default 10%
processor_fee = provider-specific       // Stripe ~2.9% + 30¢
provider_net  = gross − platform_fee − processor_fee

Booking states

pending → deposit_paid → confirmed → in_progress → completed → reviewed, with side-branches cancelled, disputed, refunded. Transitions are enforced by a Postgres trigger and mirrored in admin_audit_log.

Invoices

Format: INV-YYYYMMDD-<seq>. PDF is rendered inside generate-booking-receipt and stored in the private receipts bucket; the UI serves signed URLs valid for 1 hour.

Withdrawals

Providers request payouts from their wallet; admins approve. The approval writes a wallet_transactions debit and an off-platform withdrawals row with the payout reference.

10. Realtime & Notifications

Channel pattern

// ✅ Correct — register handlers BEFORE subscribing.
const channel = supabase.channel(`chat-${conversationId}-${uuid}`)
  .on('postgres_changes', { event: '*', schema: 'public', table: 'messages',
       filter: `conversation_id=eq.${conversationId}` }, handleMessage)
  .subscribe();

return () => supabase.removeChannel(channel);
  • Always use a unique channel id (useRef(crypto.randomUUID())) to prevent collisions across mounts.
  • Never call .on() after .subscribe() — handlers silently drop.
  • Clean up in the effect return; leaks show up as duplicate toasts.

Notification channels

The dispatch-notification-channels edge function fans a single notification out to any enabled channel per-user: in-app toast, email (Resend), and Web Push. Preferences live in notification_preferences.

Push notifications

Subscription happens in usePushNotifications() — VAPID key from env, subscription persisted in push_subscriptions, and delivery goes through send-push.

15. Internationalization & Accessibility

i18n

  • Powered by react-i18next. Bundles under src/i18n/<locale>/*.json.
  • Default locale: English. Norwegian (no) shipped.
  • Currency & date formatting via Intl APIs — no third-party.

Accessibility

  • Every interactive element has an accessible name (aria-label or visible text).
  • Color contrast ≥ 4.5:1 for body, ≥ 3:1 for large text — verified with the a11y test suite.
  • Focus rings preserved (never outline: none without a replacement).
  • Semantic headings via <Heading level={n}>; only one h1 per page.
  • Modals trap focus and restore it on close (Radix Dialog).

16. Performance

  • Route-level code splitting via React.lazy — initial JS < 180 KB gzip.
  • React Query with 60s stale time on read-heavy screens; realtime subscriptions invalidate on the fly.
  • Image optimization — WebP with lazy loading; Supabase transform URLs used for avatars & portfolio.
  • Bundle guardrail — public/bundle-size-history.json tracks each release; CI fails on > 10% growth.
  • DB indexes on bookings(status, provider_id), messages(conversation_id, created_at), vendor_services(category_id, is_active).

17. Security Model

  • RLS everywhere. No public-schema table ships without an RLS policy AND a GRANT.
  • Roles table. Roles never live on profiles — that would allow privilege escalation via profile updates.
  • Security-definer functions (has_role, is_admin) run with owner privileges to avoid recursive RLS.
  • No client-side admin checks. UI gates are cosmetic — the database is the source of truth.
  • Audit log. admin_audit_log receives triggers from payments, bookings, jobs, verifications, disputes, sponsorships.
  • File uploads validated by size + MIME on the client and re-checked in the storage policy.
  • Webhooks verify signatures (Stripe, PayPal) before mutating state.

21. Troubleshooting

"Permission denied for table" from PostgREST
The table is missing a GRANT. Every public-schema table needs explicit grants to authenticated / service_role.
Realtime channel never fires
Register all .on() handlers before .subscribe(), and use a unique channel id.
Sticky header hides when a dropdown opens
Fixed in v2.1.0 — Radix scroll-lock is neutralized in index.css and header is z-[60].
"Unsupported provider" on Google login
Enable Google provider in the Lovable Cloud auth settings.
"Invalid Refresh Token" after long idle
Session expired past the 30-day refresh window. Sign the user out and back in — a fresh session hydrates cleanly.
Push notifications don't arrive on iOS
iOS requires the app to be installed as a PWA (Add to Home Screen) and Web Push enabled in Settings → Notifications.
Realtime works locally, not in production
Check that the Realtime extension is enabled for the table (alter publication supabase_realtime add table <name>).
PDF receipts return 403
Signed URL expired (1h TTL). Re-fetch via the invoice detail endpoint to get a fresh URL.

22. FAQ

Can I rebrand the platform?
Yes — every color, logo, and copy block is editable. Per-tenant branding overrides primary/accent at runtime.
Does it support multi-tenancy?
Path-based multi-tenancy is included with RLS-isolated data per tenant.
What payment providers are supported?
Stripe and PayPal scaffolding with 25% deposit logic and commission tracking.
Is the source code editable?
Yes — full TypeScript source ships in the repo.
Can I swap Supabase for another backend?
Technically yes — data access is centralized under src/lib/api/. Practically it's a large rewrite because RLS, edge functions, and realtime channels are all Supabase-native.
How do I add a new user role?
Extend the app_role enum in a migration, update ProtectedRoute/usePermission, add a dashboard route, and write RLS policies referencing has_role(auth.uid(), '<new-role>').
Where do I add a new email template?
Insert a row into email_templates via the admin UI or migration; reference the template key from the calling edge function.
How do I localize the app for a new language?
Duplicate src/i18n/en/ to src/i18n/<code>/, translate the JSON keys, and add the language to the locale switcher.
